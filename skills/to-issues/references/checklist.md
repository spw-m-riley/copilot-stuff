# Issue breakdown checklist

Use this checklist before treating an issue breakdown as ready to publish or hand off.

## Slice quality

- Each issue is a thin vertical slice, not a layer-based task.
- Each issue is independently understandable and grabbable.
- Each issue can be demoed or verified on its own.
- Each issue has concrete acceptance criteria.
- Each issue title describes a capability or behavior, not a vague activity bucket.

## Dependency quality

- Blockers represent real sequencing needs, not convenience.
- Dependency chains are as shallow as the work allows.
- There are no circular blocker relationships.
- Blocker issues are created or planned before dependent issues.

## Classification and publication

- Each issue is marked `AFK` or `HITL` for a specific reason.
- `AFK` is the default unless a human decision is genuinely required.
- The user has approved the granularity and dependency structure before publication.
- Parent issue references, labels, or tracker fields are applied only when verified.
- If publication is blocked, the draft issue bodies are still complete and usable.
